01

Skill이 대체 뭐야?

한 줄 요약: Claude에게 주는 전문 매뉴얼이다.

카페 알바생 비유로 이해하기

새로 온 카페 알바생(Claude)은 머리가 아주 좋지만, 우리 가게만의 레시피는 모릅니다.

  ☕ "아메리카노 만들어줘" → 기본적으로 잘 만듦
  📋 "우리 가게 시그니처 라떼 만들어줘" → 레시피 없으면 자기 맘대로 만듦

이때 Skill = 우리 가게 레시피 매뉴얼입니다.
"원두는 이거, 우유 온도는 65도, 라떼아트는 하트 모양" — 이렇게 적어놓으면, 알바생은 매번 완벽하게 시그니처 라떼를 만들 수 있죠.

📄

본질은 텍스트 파일

Skill의 핵심은 SKILL.md라는 마크다운 파일 하나입니다. 거기에 "이럴 때 이렇게 해라"라고 적어둔 것이죠.

🔌

자동으로 작동

한 번 설치하면 관련 요청이 올 때 Claude가 알아서 해당 Skill을 읽고 따릅니다. 매번 지시할 필요 없어요.

🧩

무한 확장 가능

PPT 만들기, 코드 리뷰, 이메일 작성, 브랜드 가이드 적용 등 원하는 건 뭐든 Skill로 만들 수 있습니다.

Skill의 구조

skill-name/
my-awesome-skill/ ├── SKILL.md ← 핵심! 이것만 있으면 됨 │ ├── YAML 메타데이터 (이름, 언제 발동할지) │ └── 마크다운 지시문 (실제 작업 순서) │ └── 선택 사항들/ ├── scripts/ ← 자동화 스크립트 ├── references/ ← 참고 문서 └── assets/ ← 템플릿, 폰트, 이미지
02

왜 다들 Skill을 만들어 쓸까?

핵심 이유는 하나입니다 — 매번 같은 말 반복하기 싫으니까.

❌ Skill 없이 매번 하는 짓
"PPT 만들어줘. 아, 그리고 폰트는 Pretendard로, 색상은 #1a1a2e를 베이스로, 각 슬라이드에 페이지 번호 넣고, 제목은 28px Bold로, 본문은 16px Regular로..." → 매 대화마다 20줄씩 복붙 🤦
✅ Skill 만든 후
"PPT 만들어줘" → Claude가 Skill을 자동으로 읽음 → 폰트, 색상, 레이아웃 다 알아서 적용 → 매번 완벽하게 동일한 품질 ✨ 한 줄이면 끝! 🎉
🔄

일관성

Skill을 쓰면 Claude가 매번 같은 수준의 결과물을 만듭니다. 어제 잘했다가 오늘 이상하게 만드는 일이 없어지죠. 회사 브랜드 가이드, 코드 스타일 같은 걸 정해두면 흔들림이 없습니다.

⚡

속도

복잡한 지시사항을 매번 입력하면 하루에 30분은 날립니다. Skill에 한 번 적어두면 그 시간이 0이 됩니다. 하루에 10번 PPT 만드는 사람이라면? 인생이 바뀝니다.

🎯

품질 상한선 돌파

프롬프트만으로는 한계가 있습니다. Skill에 구체적인 스크립트와 템플릿, 참고 자료까지 넣으면 Claude의 결과물이 "프로 수준"으로 점프합니다.

👥

팀 공유

내가 만든 Skill을 .skill 파일로 패키징해서 팀원들에게 공유할 수 있습니다. "우리 팀은 이렇게 일한다"를 코드처럼 버전 관리할 수 있죠.

🎯
핵심 포인트: Skill은 "Claude를 내 전용 비서로 커스터마이징하는 도구"입니다. 프롬프트가 1회용 지시라면, Skill은 영구적인 업무 매뉴얼입니다.

04

나에게 필요한 Skill 찾는 법

"나도 Skill 만들어야 하나?" — 이 질문에 답하는 프레임워크입니다.

🤔 나에게 Skill이 필요할까?
✅ Skill이 필요한 상황

· 같은 형식의 결과물을 반복적으로 만들 때
· Claude 결과물의 품질이 들쑥날쑥할 때
· 팀에서 동일한 규칙을 공유해야 할 때
· 매번 긴 프롬프트를 복사해서 붙여넣을 때
· 특정 도구/라이브러리 사용법을 정해야 할 때

⏭️ 아직 필요 없는 상황

· 일회성 작업을 할 때
· 매번 다른 형식이 필요할 때
· 기본 Skill로 충분히 잘 될 때
· Claude의 기본 응답에 만족할 때

💡 Skill 아이디어 찾는 3가지 방법

🔁

반복 패턴 관찰

일주일간 Claude와 대화하며 "또 같은 말 하고 있네"라는 순간을 기록하세요. 그 반복이 곧 Skill 후보입니다.

😤

불만 포인트 수집

"Claude가 왜 맨날 이걸 빼먹지?" — 그 불만을 적어두세요. SKILL.md에 "반드시 X를 포함하라"고 쓰면 해결됩니다.

📋

체크리스트 전환

이미 쓰고 있는 작업 체크리스트가 있다면? 그걸 그대로 SKILL.md로 변환하면 됩니다.

🏷️ 인기 Skill 카테고리

📊 프레젠테이션 스타일 ✍️ 글쓰기 톤 & 스타일 💻 코드 리뷰 규칙 📧 이메일 포맷 🎨 브랜드 가이드라인 📱 앱 개발 워크플로우 📝 문서 템플릿 🧪 테스트 작성 규칙 🔍 코드 분석 패턴 📈 리포트 생성
05

Skill 만드는 법

생각보다 쉽습니다. 결국 텍스트 파일 하나 쓰는 겁니다.

1

무엇을 시킬 건지 정하기

Skill은 "Claude가 이 상황에서 이렇게 행동해라"라는 지시서입니다. 먼저 다음 3가지를 정리하세요.

  • 목적: 이 Skill이 Claude에게 무엇을 시키나? (예: "우리 회사 스타일로 기술 블로그 작성")
  • 트리거: 언제 이 Skill이 발동되어야 하나? (예: "블로그", "기술 포스트"라고 말할 때)
  • 결과물: 어떤 형태의 결과물이 나와야 하나? (예: "마크다운 파일, 2000자 이내")
2

SKILL.md 파일 작성하기

이것이 전부입니다. 아래 템플릿을 따라 작성해보세요.

SKILL.md
--- name: my-blog-writer description: 우리 회사 기술 블로그를 작성하는 Skill. 사용자가 "블로그 써줘", "기술 포스트 작성", "블로그 글 만들어줘" 등을 말하면 이 Skill을 사용한다. --- # 기술 블로그 작성 가이드 ## 글의 구조 모든 블로그 글은 다음 구조를 따른다: 1. 제목 — 호기심을 유발하는 질문형 또는 숫자 포함 2. 도입 — 독자의 문제 상황을 공감하는 1~2문장 3. 본문 — 소제목으로 나누고, 코드 예시 필수 4. 마무리 — 핵심 요약 + 다음 단계 제안 ## 톤 & 스타일 - 반말(~해요체) 사용 - 문장은 짧고 명확하게 - 이모지는 소제목에만 1개씩 - 전문 용어는 처음 등장 시 괄호로 설명 ## 반드시 포함할 것 - 실행 가능한 코드 예시 최소 2개 - "왜 이게 중요한가" 섹션 - SEO를 위한 메타 설명 (160자)
📐
작성 팁: SKILL.md의 description 부분이 매우 중요합니다. Claude가 "이 Skill을 쓸지 말지"를 description만 보고 판단하기 때문이에요. 최대한 구체적으로, 어떤 키워드가 나오면 이 Skill을 써야 하는지 적어두세요.
3

폴더 구조 만들기

가장 간단한 경우 파일 하나면 됩니다. 복잡한 Skill은 보조 파일을 추가합니다.

📦 심플 버전 (이것만으로 충분!)
my-blog-writer/ └── SKILL.md
📦 풀 버전 (필요할 때만)
my-blog-writer/ ├── SKILL.md ├── scripts/ │ └── seo-checker.py ├── references/ │ └── style-guide.md └── assets/ └── template.md
4

테스트하기

만든 Skill이 잘 작동하는지 확인합니다. 2~3개의 테스트 프롬프트를 던져보세요.

테스트 예시

프롬프트 1: "React의 useEffect에 대한 기술 블로그 써줘"
프롬프트 2: "SwiftUI의 @State를 설명하는 블로그 포스트 만들어줘"
프롬프트 3: "초보자를 위한 Git 사용법 블로그 작성"
🔄
반복 개선: 결과가 마음에 안 들면? SKILL.md를 수정하고 다시 테스트. 이걸 3~4번 반복하면 꽤 좋은 Skill이 됩니다. 완벽할 필요 없어요 — 쓰면서 다듬으면 됩니다.
5

설치 & 공유하기

완성된 Skill을 설치하고, 필요하면 팀에 공유합니다.

Claude Code CLI
# Skill 설치 (로컬) claude install skill ./my-blog-writer # .skill 파일로 패키징 (공유용) python -m scripts.package_skill ./my-blog-writer → my-blog-writer.skill 파일 생성! # 팀원이 설치하려면 claude install skill ./my-blog-writer.skill
💬
Claude.ai에서도 가능: Claude.ai 대화 중에 "이 워크플로우를 Skill로 만들어줘"라고 말하면 Claude가 SKILL.md를 생성하고 패키징까지 도와줍니다.

🎓 실전 예시: iOS 개발자를 위한 Skill

예를 들어, SwiftUI 앱 개발에 특화된 코드 리뷰 Skill을 만든다면:

swiftui-code-review/SKILL.md
--- name: swiftui-code-review description: SwiftUI 코드 리뷰 및 개선 Skill. 사용자가 "코드 리뷰해줘", "이 SwiftUI 코드 개선해줘", "Swift 코드 봐줘" 등을 말하면 이 Skill을 사용한다. 코드 품질, 성능, 아키텍처를 종합적으로 검토한다. --- # SwiftUI 코드 리뷰 가이드 코드를 리뷰할 때 다음 순서로 검토한다: ## 1. 아키텍처 점검 - MVVM 패턴을 제대로 따르고 있는가? - View에 비즈니스 로직이 섞여있지 않은가? - @StateObject vs @ObservedObject 사용이 적절한가? ## 2. 성능 점검 - 불필요한 View 리렌더링이 없는가? - .task와 .onAppear 사용이 적절한가? - 큰 리스트는 LazyVStack을 쓰고 있는가? ## 3. 리뷰 포맷 리뷰 결과는 반드시 이 형식으로 작성: ### 🟢 잘한 점 (2~3개) ### 🟡 개선 제안 (구체적 코드 포함) ### 🔴 반드시 수정 (버그/크래시 위험)
06

자주 묻는 질문

처음 접하는 분들이 가장 많이 궁금해하는 것들

네, 가능합니다! Skill의 핵심은 마크다운(.md) 파일인데, 이건 그냥 텍스트 파일입니다. "제목은 ##으로 쓴다" 정도만 알면 됩니다. 코드를 하나도 몰라도, "이 상황에서 이렇게 해라"를 글로 적을 수 있으면 Skill을 만들 수 있습니다. 사실 Claude에게 "내가 원하는 워크플로우를 Skill로 만들어줘"라고 말하면 Claude가 대신 만들어주기도 합니다.
System Prompt는 대화 전체에 적용되는 "기본 성격"이고, Skill은 특정 작업에만 발동되는 "전문 매뉴얼"입니다. 예를 들어, System Prompt로 "항상 한국어로 답해"라고 설정하면 모든 대화에 적용됩니다. 하지만 "PPT 만들 때는 이 폰트를 써라"는 PPT를 만들 때만 필요하죠 — 이런 건 Skill이 적합합니다. 또한 Skill은 SKILL.md 파일에 보조 스크립트, 템플릿, 참고 문서까지 묶을 수 있어서 훨씬 강력합니다.
이름과 description만 항상 로딩되고, SKILL.md 본문은 필요할 때만 읽습니다. 따라서 Skill이 많아도 description이 짧고 명확하면 성능 영향은 거의 없습니다. 다만 description이 너무 광범위하면 관계없는 Skill이 자꾸 발동될 수 있으니, 트리거 조건을 명확하게 쓰는 게 중요합니다.
네! .skill 파일만 있으면 바로 설치할 수 있습니다. 커뮤니티에서 공유되는 Skill을 다운받아 설치하거나, 팀원이 만든 Skill을 공유받아 사용할 수 있습니다. 마치 앱을 설치하듯이 간편하게 추가하고 제거할 수 있어요.
1. description이 너무 짧거나 모호함 — Claude가 Skill을 쓸지 결정하는 핵심이 description인데, "블로그 작성 도구"처럼 대충 쓰면 잘 발동되지 않습니다. "사용자가 '블로그', '기술 포스트', '글 써줘'라고 말하면 이 Skill을 사용한다" 수준으로 구체적으로 적어야 합니다.

2. 지시가 너무 추상적 — "좋은 글을 써라" 대신 "첫 문장은 독자의 고민을 질문으로 시작한다", "소제목은 3~5개" 같이 구체적으로 써야 합니다.

3. 테스트를 안 함 — 만들고 바로 실전 투입하면 예상 못한 결과가 나옵니다. 최소 2~3개의 테스트 프롬프트로 확인 후 개선하세요.
Claude.ai에서도 Skill 시스템이 작동합니다. 실제로 Claude.ai의 파일 생성 기능(docx, pptx, xlsx 등)이 바로 내장된 Skill을 통해 구현되어 있습니다. 다만 커스텀 Skill의 설치 방법과 관리 방식은 Claude Code(CLI)와 다를 수 있으므로 최신 문서를 확인하세요.

요약하면 이겁니다

Skill = Claude에게 주는 영구적 업무 매뉴얼
만드는 법 = SKILL.md 파일 하나 작성
핵심 가치 = 반복 제거 + 일관된 품질
